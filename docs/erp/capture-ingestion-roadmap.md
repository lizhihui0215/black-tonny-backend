# ERP Capture 全量导入路线图

## 1. 最终目标

在当前合法账号 / 角色可见范围内，把所有可读报表与查询数据：

1. 研究透接口语义与参数边界
2. 通过纯 HTTP 稳定重放
3. 准确、可追溯地落入 `black_tonny_capture`
4. 再按可信度分层投影到 `black_tonny_serving`

这条路线不追求“一次性全部接完”，而是坚持：

- 先把当前已知全部读接口做成完整风险地图
- 再按成熟度分批准入 capture
- 最后再决定是否进入 serving / 上层 API 主链

统一状态面板以 [ERP API 成熟度总览](./api-maturity-board.md) 为准；本文件只定义推进顺序、准入门槛和落库规则。

---

## 2. 推进原则

### 2.1 三层职责不变

- `capture`
  - 只存原始或近原始 payload
  - 必须保留请求参数、批次号、账号上下文、抓取时间
  - 是后续回放、对账、纠错的稳定底座
- `serving`
  - 继续只做可演进投影层
  - 只有已经通过准入的路线，才允许逐步投影进去
- 上层 API / 页面
  - 只消费已准入的 serving 数据
  - 不直接读未确认的研究路线

### 2.2 全域门槛优先于路线准入

在任何路线进入正式 `capture` 主链之前，必须先满足一个全域门槛：

1. 当前账号可见菜单树已经完成覆盖审计
2. 所有可点击页面都已经被分类到：
   - `covered`
   - `visible_but_untracked`
   - `visible_but_failed`
   - `container_only`
3. 当前已知全部读路线都至少达到 `已基线`
4. 每条路线都已明确：
   - 分页语义风险
   - 枚举/范围切换风险
   - 主源候选 / 对账源 / 结果快照 / 未采纳 分类
   - 当前可信度等级
   - 下一步动作

只有全域门槛达成后，才允许开始评估单条路线的 `capture` 准入。

任何路线要进入正式 `capture` 主链，还必须继续满足：

1. 页面研究已确认接口语义
2. 纯 HTTP 可稳定重放
3. 关键参数已分清“视图切换”还是“数据范围切换”
4. 分页语义已确认
5. 至少有一条对账或证据闭环能解释主要指标
6. 当前没有高优先级 blocker

未满足时，只能保持：

- `研究中`
- `对账源`
- `结果快照`
- `未采纳`

### 2.3 精准性优先于速度

后续所有抓取推进都遵循：

- 不把同一业务域里不同粒度的路线直接混算
- 不把结果快照接口误当成事实源
- 不把“页面能查到”直接当成“已经适合进主链”
- 不因为想尽快全量抓取，就跳过分页、枚举、范围语义确认

## 3. 路线状态到动作映射

状态板里的每个阶段，都对应固定动作：

- `已发现`
  - 动作：补页面基线研究
  - 目标：确认页面入口、主接口、基础参数形状
- `已基线`
  - 动作：补单变量深挖
  - 目标：识别分页、枚举、组织范围、DIY 上下文
- `已单变量`
  - 动作：补纯 HTTP 回证
  - 目标：把浏览器研究结论回落到正式抓取参数
- `已HTTP回证`
  - 动作：收口 blocker，评估 capture 准入
  - 目标：决定是否进入正式 `capture_endpoint_payloads`
- `已准入主链`
  - 动作：批量抓取、回归监控、异常打标
  - 目标：稳定成为 capture 主链输入

只有到 `已准入主链`，才允许把该路线纳入持续性抓取任务。

## 4. 分阶段推进顺序

### Phase 0：冻结正式入库，先完成全域风险地图

目标：先把“当前账号可见全域”做成完整风险地图，再决定哪些路线值得分批进入正式 `capture`。

执行要求：

- `capture` 不接新的正式主链路线
- `serving` 不接任何新路线
- 浏览器研究和纯 HTTP 回证继续只产出：
  - 本地证据
  - `analysis/*.json`
  - 文档结论
- 所有已知路线都必须至少完成：
  - 页面基线
  - 过滤条件识别
  - 分页存在性判断
  - 枚举/范围参数识别
  - 主源/快照分类
- 菜单覆盖审计必须完成：
  - 导出当前账号完整菜单树
  - 逐个可点击页面归类成 `covered / visible_but_untracked / visible_but_failed / container_only`
  - 对 unknown page 生成占位 route，而不是静默漏页

Phase 0 完成标准：

1. 当前账号可见菜单覆盖审计已完成
2. 所有可点击页面都已分类，不再存在“未审计页面”
3. 状态板里全部已知路线都达到明确分类与下一步
4. 全域门槛达成前，不允许任何路线开始正式写入 `capture`

### Phase 1：全域风险地图完成后，先执行销售首批 capture 准入

目标：把销售域从“已研究透”推进到“首批 capture 可持续留痕”，但暂不放开 `serving`。

优先路线：

- `SelSaleReport`
  - 定位：订单头候选源
  - 当前结论：
    - `sale_no` 已确认是唯一主关联键
    - 订单头唯一性当前已通过
    - 下一步是按 `sales_documents_head` 执行首批 capture 准入并观测回归指标
- `GetDIYReportData(E004001008_2)`
  - 定位：明细行候选源
  - 当前结论：
    - 已可按 `sale_no` 分流正常销售明细与逆向明细
    - 下一步是按 `sales_document_lines` 与 `sales_reverse_document_lines` 分别留痕
- `SelDeptSaleList`
  - 定位：研究 / 对账源
  - 当前结论：
    - 常规结束日期窗口下 `edate` 已确认是 `same_dataset`
    - 只有压到单日时才会退化成 edge case
    - 继续保留为研究 / 对账源，不进入事实主源候选

Phase 1 完成标准：

1. `sales_documents_head` 与 `sales_document_lines` 能在同一批次下稳定写入 capture
2. `sales_reverse_document_lines` 被单独分流为研究留痕路线，不混入正常销售主链
3. `SelDeptSaleList` 固定为研究 / 对账源
4. 销售首批 capture 具备固定回归指标和异常阈值，但 `serving` 继续冻结

### Phase 2：库存域收口并形成第二批 capture 主链候选

目标：明确库存事实路线的正式抓取参数集。

优先路线：

- `库存明细统计 / SelDeptStockWaitList`
  - 当前 blocker：
    - `stockflag=1/2` 是否正式并入
    - 分页终止条件仍需收口
- `出入库单据 / SelOutInStockReport`
  - 当前 blocker：
    - `type`
    - `doctype`

保留结果快照定位：

- `库存总和分析-*`
- `库存多维分析`
- `库存零售统计`
- `进销存统计`
- `日进销存`

Phase 2 完成标准：

1. `stockflag` 正式抓取策略固定
2. `type / doctype` 的语义固定
3. 库存事实路线能形成稳定 capture 批次

### Phase 3：会员 / 储值 / 流水单据完成最小闭环

目标：从“已基线”推进到“已 HTTP 回证”，但不抢在销售、库存之前进主链。

优先路线：

- `会员中心 / SelVipInfoList`
- `储值卡明细 / GetDIYReportData`
- `每日流水单 / SelectRetailDocPaymentSlip`

这三条都先做：

1. 单变量探测
2. 纯 HTTP 回证
3. 范围与分页确认

Phase 3 完成标准：

- 至少形成各域一条“可评估 capture 准入”的候选路线

### Phase 4：从“研究完成”切到“capture 持续化”

当路线通过准入标准后，再做两件事：

1. 把它纳入正式 `capture_batches + capture_endpoint_payloads`
2. 为它定义固定回归指标和异常阈值

这一阶段不再新增新路线，而是把已准入路线变成稳定抓取任务。

## 5. Capture 落库规则

### 5.1 必留字段

所有正式进入 capture 的路线，都必须稳定保留：

- `capture_batch_id`
- `source_endpoint`
- `page_no` / `page_cursor`
- `request_params`
- `payload_json`
- `checksum`
- `pulled_at`
- 当前账号 / 组织上下文

### 5.2 批次要求

每一轮正式抓取都必须可追溯：

- 同一批次内，时间范围和账号上下文固定
- 同一条路线，分页请求都挂到同一个 `capture_batch_id`
- 如果抓取中断，必须能明确知道断在哪一页、哪一个参数组合

### 5.3 异常处理

只要出现以下情况之一，就不能自动视为“成功可用”：

- 行数突变
- 单据数突变
- 金额突变
- 分页提前结束
- 枚举值返回结构与预期不一致
- 关键对账指标从“一致/可接受差异”退化到“差异待解释”

这类批次应：

- 写入 capture
- 打标异常
- 不自动推进 serving 主链

---

## 6. 数据精准与稳定规则

### 6.1 主源唯一

同一个经营指标只能认一条主源路线。

例如销售域：

- 正常销售订单头与明细行可以并存
- `sales_reverse_document_lines` 只保留为逆向研究留痕路线
- 但 `SelDeptSaleList` 在当前阶段只能做研究 / 对账
- 不能把它和 `SelSaleReport`、`GetDIYReportData(E004001008_2)` 同时作为主源

### 6.2 固定回归指标

每条准入 capture 的路线都要固定一组回归指标：

- 行数
- 单据数
- 件数
- 金额
- 与上一批偏差阈值

### 6.3 页面研究与 HTTP 回证必须一致

如果页面研究和纯 HTTP 回证不一致：

- 一律标记为 `待解释`
- 不准直接接入 capture 主链

## 7. 后续执行规则

从现在开始，后续每轮推进都按这个顺序选任务：

1. 先看 [ERP API 成熟度总览](./api-maturity-board.md)
2. 先确认全域门槛是否已达成：
   - 如果未达成，优先补 `research_map_complete = false` 的路线
   - 如果已达成，再从 `mainline_ready = 否` 且 `next_action` 最明确的高价值路线里选下一条
3. 高价值域优先级固定：
   - 销售主源候选
   - 库存主源候选
   - 会员 / 储值 / 流水单据主源候选
   - 结果快照路线
4. 做完一轮后，先刷新状态板，再决定下一轮

也就是说，后续推进的节奏固定为：

`研究 -> 回证 -> 准入 -> capture -> 回归监控`

而不是：

`先抓全量 -> 发现不准 -> 再返工`
