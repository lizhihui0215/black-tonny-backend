# AI 协作低 Token 操作手册

这份手册用于降低 `black-tonny-backend` 仓库里的 AI 协作会话 token 消耗。

它是执行辅助，不替代 [../../AGENTS.md](../../AGENTS.md) 里的优先级、读顺序和仓库规则。

## 适用场景

- Codex / ChatGPT / Claude / Gemini 等 AI 编码协作
- repo-local 文档导航、问题排查、方案分析
- MCP、Playwright、研究脚本等工具型任务

不适用场景：

- 正式运行时模型调用成本审计
- 业务口径定义
- `capture` / `serving` 主链准入规则

这些内容仍以对应业务文档为准。

## 最小读法流程

1. 先用 [../../AGENTS.md](../../AGENTS.md) 里的 task routing 判断任务类型。
2. 默认只打开 1 到 2 个最相关入口文档，不先扫全量入口。
3. 先读相关章节或相关片段，再决定是否需要整篇阅读。
4. 只有当任务触达契约、边界、数据流或 source-of-truth 规则时，才继续扩读深层文档。

## 搜索与阅读默认值

- 搜索优先限定在最可能相关的目录和关键字，不默认做 workspace 级宽搜。
- 找代码入口优先用精确 `rg`，例如按 route、service、schema 名称或 endpoint 关键字缩小范围。
- 阅读大文件时优先先找标题、段落或关键函数，再按片段读取。
- 工具输出只保留能支撑当前判断的最小证据，不把长搜索结果整段带进后续多轮。
- 如果已经确认某条事实，后续优先引用摘要，不重复贴原文。

## 事实缓存模板

每轮探索后，建议沉淀 3 到 5 行固定摘要：

```text
已确认：
- …

已排除：
- …

下一步：
- …
```

建议规则：

- `已确认` 只保留当前已验证的仓库事实
- `已排除` 只写已经看过且可停止继续追的方向
- `下一步` 只写一个最直接的后续动作

## 何时必须扩读

出现下面任一情况时，不应继续只靠最小读法：

- 要修改公共 API 契约或响应结构
- 要调整 runtime ownership、frontend-backend boundary 或数据库边界
- 要把 tooling / research 行为推进到正式链路
- 当前证据不足以判断 source-of-truth 文档归属

## 不建议的做法

- 一上来就整篇打开所有入口文档
- 默认执行跨整个 workspace 的宽范围搜索
- 为了进度同步重复转述同一结论
- 在多轮对话里反复粘贴同一段长输出
- 还没确认边界就开始扩展到无关 area docs

## 简单验收口径

如果这套读法生效，通常会出现下面这些变化：

- 初始探索读取的文件数更少
- 很少出现无边界的大范围搜索
- 同一事实不会被重复读取和重复复述
- 在不牺牲规则正确性的前提下，更快定位到要改的入口
