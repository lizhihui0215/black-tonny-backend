# MCP 使用说明

## 1. 这份文档的作用

这份文档只解释 MCP 在 `black-tonny-backend` 仓库里的使用边界。

它回答的是：

- MCP 在这里是干什么的
- MCP 配置通常放在哪里
- 在 AI 会话里怎么调用
- 它的权限边界是什么
- 常见误解有哪些

它不回答 backend 运行时如何加载 MCP，因为这个仓库本身不依赖 MCP 才能运行。

## 2. MCP 在本仓库里的定位

当前定位固定为：

- AI 会话辅助工具
- 文档检索、浏览器操作、上下文读取的扩展能力
- 研究和协作效率工具

不是：

- FastAPI 运行时依赖
- `capture` 任务运行时依赖
- `serving` 投影运行时依赖
- CI/CD 必需组件

换句话说：

- 没有 MCP，这个仓库依然可以运行
- 有 MCP，主要是帮助 AI 会话更高效地读文档、操作浏览器、做研究

## 3. 配置位置

MCP 配置通常不放在这个仓库里。

常见位置是：

- AI 客户端或 CLI 的用户级配置目录
- 本机开发环境自己的工具配置

在当前仓库里，MCP 只会通过文档引用和使用约定出现，不会作为 repo 内受版本管理的正式运行配置。

因此默认规则是：

- 不把 user-scoped MCP 配置误写成仓库运行时依赖
- 不把“我本机能用的 MCP”写成团队必须具备的前提

## 4. 调用方式

在 AI 会话里，MCP 通常用于这几类事情：

- 读取本地资源或远端文档
- 查询结构化上下文
- 辅助浏览器研究
- 补足官方文档检索能力

在这个仓库里，MCP 的典型使用场景包括：

- 辅助读取 OpenAI 官方文档
- 辅助浏览器自动化或页面研究
- 辅助列出或读取外部上下文资源

如果任务只是修改业务代码或文档，并不要求你必须使用 MCP。

## 5. 权限边界

默认边界：

- MCP 只能作为研究和协作辅助
- 不能把 MCP 当作正式抓取主链的一部分
- 不能把 MCP 结果直接当作业务事实源
- 不能绕过已有的文档、测试和运行时边界

对于 backend 尤其要注意：

- `capture` 主链的准入判断不能依赖 MCP
- `serving` 投影的准入判断不能依赖 MCP
- `/api/dashboard/summary` 的契约和实现不能建立在某个 MCP 是否可用之上

## 6. 什么时候该用，什么时候不该用

适合用：

- 你需要补充外部官方资料
- 你需要快速操作浏览器做研究
- 你需要从 AI 会话里读取额外结构化上下文

不适合用：

- 你只是做仓库内普通代码修改
- 你在定义正式业务口径
- 你在决定某条 `capture` 路线是否可准入
- 你在写正式运行时依赖说明

## 7. FAQ

### Q1. MCP 是这个仓库的必需依赖吗？

不是。

它只是 AI 会话辅助工具，不是 backend 运行时依赖。

### Q2. MCP 能不能替代 `capture` 研究或纯 HTTP 回证？

不能。

MCP 只能辅助研究，不能替代正式抓取链和主链准入判断。

### Q3. MCP 配置是不是应该提交到仓库？

默认不应该。

除非未来明确引入 repo-managed MCP 配置，否则都按 user-scoped 工具处理。

### Q4. 如果一个任务只有 MCP 能做，是否可以把它写进业务文档？

可以记录“研究时可选使用 MCP”，但不能把它写成正式主链前提。
